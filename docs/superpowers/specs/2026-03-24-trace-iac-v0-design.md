# TRACE IaC Agent v0 Design

Date: 2026-03-24
Status: Draft approved in conversation, written for review
Scope: v0 runtime only

## 1. Summary

This document defines the v0 architecture for a staged agent that transforms user intent into a topology intermediate representation (TGraph IR). The v0 system uses three isolated agent stages:

1. `ground`
2. `logical`
3. `physical`

After `physical`, the runtime calls a non-agent translate stub. Translation is intentionally deferred and is not part of v0 design scope.

The design goal is not a full IaC generator yet. The design goal is a reliable, observable, stage-isolated runtime with:

- explicit artifact flow across stages
- repairable `tgraph.json` generation in `logical` and `physical`
- optional Python validator scripts generated by the agent in `logical` and `physical`
- LangSmith tracing for stage-level and round-level observability
- local run checkpoints and resume support

## 2. Goals

### 2.1 In scope

- Build an agent runtime in Python on top of LangChain
- Enforce stage isolation through explicit artifact selection
- Support three agent stages only: `ground`, `logical`, `physical`
- Normalize compact node declarations such as `PLC[1..6]`
- Treat `logical_constraints` and `physical_constraints` as natural-language constraints
- Allow constraints to target either specific node IDs or the whole topology
- Let `logical` and `physical` produce:
  - checkpoints
  - a `tgraph.json`
  - an optional Python validation script
- Validate `tgraph.json` through a runtime-controlled checkpoint runner
- Support iterative repair when validation fails
- Record execution with LangSmith and local checkpoints

### 2.2 Out of scope

- Real IaC translation
- The `close` stage
- Long-term semantic memory as a core system feature
- Multi-backend IaC generation such as Terraform and Bicep
- Production sandboxing for untrusted code execution beyond local v0 constraints

## 3. Product decisions confirmed during design

### 3.1 Stage set

The workflow for v0 is:

`ground -> logical -> physical -> translate_stub`

`close` is removed from v0.

### 3.2 Stage isolation rule

Stage isolation is defined as:

- a stage may only read artifacts explicitly declared in its `StageSpec.inputs`
- a stage may not read raw prior chat history
- a stage may not read arbitrary memory from previous stages

The isolation boundary is artifact-based, not "previous stage result object" based.

### 3.3 Ground stage output model

`ground` identifies node IDs and constraints.

- Node output exists primarily to define available node IDs
- Node kinds are not required in v0
- Compact forms such as `PLC[1..6]` are allowed in agent output for token efficiency
- The runtime expands compact declarations into canonical node ID lists

### 3.4 Constraint format

`logical_constraints` and `physical_constraints` remain natural language.

Each constraint must be anchored in one of two scopes:

- `node_ids`
- `topology`

Node-scoped constraints should reference concrete node IDs whenever possible.

### 3.5 P2 and P3 checkpoint format

`logical` and `physical` checkpoints are structured as:

- function name
- input parameters
- description

These checkpoints may optionally reference a Python script artifact generated by the agent.

### 3.6 Python validation scripts

`logical` and `physical` may generate Python scripts for validation support.

Constraints on generated scripts:

- pure Python is allowed
- network access is not allowed
- external side effects are not allowed
- runtime owns execution
- the agent never executes code directly

## 5. Core architecture

### 5.1 Architectural principle

The system is artifact-first.

The primary data plane is not free-form memory. The primary data plane is explicit artifacts produced by each stage and selected by later stages through a declared input contract.

### 5.2 Layer responsibilities

#### `app/`

Owns orchestration only.

- stage order
- state transitions
- failure handling
- checkpoint writing
- resume behavior

`app/` must not contain business reasoning.

#### `stages/`

Owns stage-specific differences only.

- prompt path
- input contract
- output schema
- guard logic
- normalization hooks where needed

#### `agent/`

Owns the stable agent abstraction.

- runtime calls only `AgentFacade`
- business code does not depend directly on LangChain APIs
- tracing integration is contained under `agent/langchain/`

#### `artifacts/`

Owns artifact storage and retrieval.

- writing versioned artifacts
- selecting declared input artifacts
- summarizing large artifacts for repair rounds

#### `validators/`

Owns validation and repair mechanics.

- execute checkpoints
- load optional Python validator scripts
- return structured validation reports
- apply patch operations to graphs

#### `tools/`

Owns capabilities callable by the agent.

- centralized registry
- stage-level tool whitelisting

## 6. Core domain contracts

### 6.1 Artifact selection

Later stages may read artifacts from multiple prior stages. This is required by the confirmed data flow.

Example intent:

- `logical` reads `ground.expanded_node_ids` and `ground.logical_constraints`
- `physical` reads:
  - `ground.expanded_node_ids`
  - `ground.physical_constraints`
  - `logical.logical_checkpoints`
  - `logical.tgraph_logical`

Recommended core contract shapes:

```python
class ArtifactSelector(BaseModel):
    stage: Literal["ground", "logical", "physical"]
    name: str
    required: bool = True

class ArtifactRef(BaseModel):
    stage: Literal["ground", "logical", "physical", "runtime"]
    name: str
    version: int
    path: str
    sha256: str
```

### 6.2 Stage specification

```python
class StageSpec(BaseModel):
    id: Literal["ground", "logical", "physical"]
    prompt_path: str
    inputs: list[ArtifactSelector]
    output_model: type[BaseModel]
    allowed_tools: list[str]
    max_rounds: int
    repair_mode: Literal["none", "patch"]
```

Expected values:

- `ground.repair_mode = "none"`
- `logical.repair_mode = "patch"`
- `physical.repair_mode = "patch"`

### 6.3 Validation report

```python
class ValidationIssue(BaseModel):
    code: str
    message: str
    severity: Literal["error", "warning"]
    scope: Literal["node_ids", "topology"]
    targets: list[str] = []
    json_paths: list[str] = []

class ValidationReport(BaseModel):
    ok: bool
    issues: list[ValidationIssue]
```

### 6.4 Stage memory

The runtime should keep only compact repair-oriented memory for `logical` and `physical`.

```python
class StageMemory(BaseModel):
    summary: str
    recent_changes: list[str]
    open_issues: list[str]
    active_artifacts: list[ArtifactRef]
```

This memory must not contain the whole `tgraph.json` unless the graph is trivially small.

### 6.5 Run state

```python
class RunState(BaseModel):
    run_id: str
    session_id: str
    current_stage: Literal["ground", "logical", "physical"] | None
    status: Literal["running", "failed", "completed"]
    artifacts: list[ArtifactRef]
```

## 7. Stage-by-stage design

### 7.1 Ground

#### Inputs

- user intent text

#### Agent output

```python
class ConstraintItem(BaseModel):
    id: str
    scope: Literal["node_ids", "topology"]
    targets: list[str] = []
    text: str

class GroundOutput(BaseModel):
    node_patterns: list[str]
    logical_constraints: list[ConstraintItem]
    physical_constraints: list[ConstraintItem]
```

#### Runtime post-processing

The runtime expands `node_patterns` into canonical node IDs.

Example:

- input: `PLC[1..6]`, `HMI1`
- derived output: `PLC1`, `PLC2`, `PLC3`, `PLC4`, `PLC5`, `PLC6`, `HMI1`

#### Ground artifacts

- `node_patterns.json`
- `expanded_node_ids.json`
- `logical_constraints.json`
- `physical_constraints.json`

### 7.2 Logical

#### Inputs

- `ground.expanded_node_ids`
- `ground.logical_constraints`

#### Outputs

- `logical_checkpoints.json`
- `tgraph_logical.json`
- optional `logical_validator.py`

Suggested schema:

```python
class CheckpointSpec(BaseModel):
    id: str
    function_name: str
    input_params: dict[str, Any]
    description: str
    script_ref: str | None = None

class LogicalOutput(BaseModel):
    logical_checkpoints: list[CheckpointSpec]
    tgraph_logical: dict
    logical_validator_script: str | None = None
```

### 7.3 Physical

#### Inputs

- `ground.expanded_node_ids`
- `ground.physical_constraints`
- `logical.logical_checkpoints`
- `logical.tgraph_logical`

#### Outputs

- `physical_checkpoints.json`
- `tgraph_physical.json`
- optional `physical_validator.py`

Suggested schema:

```python
class PhysicalOutput(BaseModel):
    physical_checkpoints: list[CheckpointSpec]
    tgraph_physical: dict
    physical_validator_script: str | None = None
```

## 8. Validation and repair loop

### 8.1 Why repair exists

`logical` and `physical` produce large graphs. Those graphs may be invalid or inconsistent. Rewriting the whole graph every round is inefficient and unstable. Therefore the runtime should validate and repair incrementally.

### 8.2 Validation execution model

The runtime executes validation, not the agent.

Execution flow:

1. load current `tgraph`
2. load checkpoint metadata
3. execute built-in validator functions
4. if a checkpoint references a generated script, load that script and call the requested function
5. aggregate results into one `ValidationReport`

### 8.3 Recommended generated script interface

Use a simple callable contract:

```python
def check(tgraph: dict, **kwargs) -> list[dict]:
    ...
```

or:

```python
def check_xxx(tgraph: dict, **kwargs) -> list[dict]:
    ...
```

`function_name` from `CheckpointSpec` selects the target function.

### 8.4 Repair inputs

When validation fails, the next agent round should receive:

- stage summary
- failed checkpoint IDs
- issue messages
- affected node IDs
- affected JSON paths
- relevant subgraph slice
- recent patch history
- recent script execution summary

The next round should not receive the full graph by default.

### 8.5 Repair outputs

The preferred repair output is patch-oriented.

```python
class GraphPatchOp(BaseModel):
    op: Literal[
        "add_node",
        "remove_node",
        "update_node",
        "add_edge",
        "remove_edge",
        "update_edge"
    ]
    path: str | None = None
    value: dict[str, Any] | None = None
    reason: str
```

Repair rounds may update:

- graph patch operations
- checkpoint metadata
- generated validator script

### 8.6 Authoritative data rule

The authoritative graph always lives in artifact storage. Prompt memory holds only summaries and repair context.

## 9. LangSmith observability design

### 9.1 Principle

LangSmith should observe reasoning flow. Artifact storage should preserve the full authoritative outputs.

Large files such as full graphs or generated Python scripts should be stored on disk and referenced in traces by metadata, path, version, and summary rather than inlined everywhere.

### 9.2 Trace hierarchy

Recommended trace structure:

1. root run: full workflow
2. child run: one per stage
3. grandchild run: one per agent round
4. sibling child runs inside a stage for:
   - validation
   - patch application

### 9.3 Metadata to attach

At minimum:

- `run_id`
- `session_id`
- `stage_id`
- `attempt`
- `input_artifact_refs`
- `output_artifact_refs`
- `repair_mode`
- `validator_ok`
- `graph_summary`
- `validation_summary`
- `prompt_fingerprint`

Recommended summaries:

- node count
- edge count
- checkpoint count
- script function count
- error count
- warning count
- changed JSON paths

### 9.4 Trace ownership

The tracing API should be hidden behind `agent/langchain/tracing.py` so the business runtime does not depend on LangChain tracing details.

## 10. Resume and checkpointing

### 10.1 Resume principle

Resume should restore from saved run state and artifacts, not from conversational context.

### 10.2 Checkpoint policy

Write checkpoints:

- before each stage starts
- after each stage succeeds
- after each failed validation cycle
- after each successful repair cycle

### 10.3 Local run layout

Each `run_id` should maintain:

- `state.json`
- `timeline.jsonl`
- numbered checkpoint snapshots
- versioned stage artifacts
- validation reports

Example:

```text
runs/<run_id>/
  state.json
  timeline.jsonl
  checkpoints/
    000_ground_started.json
    001_ground_succeeded.json
    002_logical_attempt1_failed.json
    003_logical_attempt2_succeeded.json
```

### 10.4 Supported recovery modes in v0

Two recovery modes are enough for v0:

1. continue from latest checkpoint
2. restart from a selected stage

Fine-grained arbitrary rollback is intentionally deferred.

## 11. Testing strategy

### 11.1 Unit tests

Focus on deterministic boundaries:

- node pattern normalization
- artifact selection rules
- tool allowlist filtering
- checkpoint execution
- generated script loading
- validation report normalization
- patch application
- stage summary extraction from large graphs

### 11.2 Integration tests

Core path tests:

- `ground -> logical`
- `logical` validation failure then repair
- `logical -> physical`
- `physical` reads both `ground` and `logical` outputs correctly

### 11.3 End-to-end tests

Core CLI tests:

- `run`
- `resume`
- run folder creation
- checkpoint persistence

### 11.4 LLM test strategy

CI should default to a fake or fixture-based `AgentFacade`.

The runtime should be stabilized before relying on real-model tests.

## 13. Open items deferred beyond v0

- real translation module
- final target IaC backend selection
- stronger isolation and sandboxing for generated validator code
- richer long-term memory
- cloud deployment strategy
- remote repository initialization and first push workflow

## 14. Final recommendation

The original proposed architecture is directionally correct. For v0, it should be narrowed into an artifact-first three-stage runtime with strong stage contracts, repair-oriented graph validation, and explicit observability boundaries.

This provides the best path to:

- fast implementation of the framework
- stable debugging
- clear LangSmith visibility
- future extension to translation and additional stages
